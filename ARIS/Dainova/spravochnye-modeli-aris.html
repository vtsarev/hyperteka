Разработка справочных моделей ARIS как способ снижения трудоемкости ведения корпоративной архитектуры

https://dainova.su/articles/razrabotka-spravochnyh-modeley-aris/

Данная статья посвящена использованию справочных моделей в ARIS, обеспечивающих классификацию объектов для удобства хранения и повторного использования объектов в репозитории моделей.

Модели справочников – это закрытые для редактирования модели, предоставляющие объекты бизнес-деятельности организации для использования их при описании бизнес-процессов, управление которыми осуществляется централизовано –  владельцем модели или ответственным за ведение данной модели.

Ниже представлен некоторый перечень основных справочных моделей. 

Модель «Организационная схема» – предназначена для иерархического моделирования организационной структуры предприятия и распределения функций управления между лицами, принимающими решение, и подчиненным персоналом. Данная модель в основном используется для представления следующих объектов:

  • отделы;
  • должности;
  • сотрудники;
  • группы;
  • роли;
  • местоположение.

При разработке организационной структуры между структурными элементами, как правило, устанавливаются следующие типы связей:

  • является организационным управляющим/является техническим руководителем;
  • состоит из;
  • имеет в подчинении/взаимодействует с.

Модель «Тип прикладной системы» – предназначена для моделирования ИТ-инфраструктуры организации и преимущественно используется для представления следующих данных:

  • каталог ИТ-систем;
  • структуры ИТ-системы;
  • технологической платформы ИТ-систем.

При разработке данной модели между объектами могут быть использованы разные типы связи, в зависимости от контекста, но, как правило, устанавливается связь – «объединяет».

Рисунок 3. Модель «Тип прикладной системы»

Модель «Бизнес-данные» – предназначена для создания единой методики семантического описания и учета бизнес-данных.  Эта модель содержит такие объекты, как сущность, отношение и атрибуты. Между сущностями могут быть использованы разные типы связи, в зависимости от контекста, но, как правило, устанавливается связь – «имеет связь с». Отношение между сущностями обладает степенью сложности, указывающей, сколько сущностей одного типа связаны с одной сущностью другого типа. Выделяют четыре типа отношений:

  • 1:1 – только одна сущность из второго типа может вступать в отношения с каждой сущностью из первого типа;
  • 1:n – только одна сущность из второго типа связана с каждой сущностью из первого, но n сущностей из первого типа связаны с каждой сущностью из второго типа;
  • n:1 – обратный порядок связи вышеописанного отношения;
  • n:m – несколько сущностей из второго типа связаны с одной сущностью из первого типа и наоборот.

Для упрощения описания сложных моделей рекомендовано кластерное хранение данных организации по бизнес-доменам или бизнес-областям, которые объединяют логически связанные типы сущностей и отношений.

Модель «Дерево продуктов и услуг» – предназначено для моделирования сложных продуктов и услуг, предлагаемых организацией, и содержащих несколько частей, каждая из которых имеет различный перечень составляющих. В большинстве случаев, используется для представления следующих объектов:

  • продукты, которые продает компания;
  • услуги, предлагаемые клиентам.

При разработке данного дерева между объектами, как правило, устанавливаются следующие связи:

  • состоит из;
  • имеет отношение к.

Рисунок 6. Модель «Дерево продуктов и услуг»

Модель «Дерево целей» – предназначена для иерархического моделирования целей компании, в частности, результатов деятельности компании, которые предполагается достичь, при поддержке способствующих успеху факторов и реализации новых бизнес-процессов. Как правило, данная модель используется для представления следующих данных:

  • цель компании – предполагаемые результаты деятельности;
  • критический фактор – аспекты деятельности, которые необходимо рассмотреть, чтобы достичь отдельной цели компании;
  • функция – задача, ведущая к достижению цели.

При разработке данной модели между объектами, как правило, устанавливаются следующие связи:

  • принадлежит;
  • оценивает;
  • поддерживает.

Рисунок 7. Модель «Дерево целей»

Модель «Дерево требований» – предназначена как для моделирования требований, помогающих в достижении целей, так и для моделирования требований к ИТ-системам. В большинстве случаев, используется для представления следующих данных:

  • требование – аспекты потребностей;
  • цель компании – предполагаемые результаты деятельности.

При разработке модели требований между объектами, как правило, устанавливаются следующие связи:

  • объединяет;
  • поддерживается.

Модель «Функциональность системы» – предназначена для моделирования функционала системы, автоматизирующего шаги процесса. Для упрощения описания сложных моделей рекомендовано кластерное хранение функционала по бизнес-доменам. Как правило, данная модель используется для представления следующих данных:

  • ИТ-система;
  • функциональность ИТ-системы.

При разработке данной модели между объектами могут быть использованы разные типы связи, в зависимости от контекста, но, как правило, устанавливается связь – «объединяет».

Модель «Дерево бизнес-функций» – предназначена для иерархического моделирования выполняемых функций, образующих один логический этап в рамках процесса.

При разработке данной модели между объектами могут быть использованы разные типы связи, в зависимости от контекста, но, как правило, устанавливается связь – «состоит из».

Модель «Дерево рисков» – предназначена для описания рисков, связанных с различными типами человеческих и технических ошибок, возникающих в процессе деятельности организации. Как правило, данная модель используется для представления следующих данных:

  • категория риска;
  • риск.

При разработке данной модели между объектами, как правило, устанавливаются следующие связи:

  • объединяет;
  • содержит.

Модель «Дерево документов» – предназначена для структурированного описания документов организации. В большинстве случаев используется классификация по направлениям деятельности или по типам документов. Как правило, данная модель используется для представления следующих данных:

  • устная информация;
  • документ;
  • электронный документ;
  • файл;
  • картотека;
  • модель ARIS.

При разработке данной модели между объектами, как правило, устанавливается связь – состоит из.

Также справочные модели могут создаваться автоматически на основе данных из сторонних систем, импортируемых при помощи специальных инструментальных средств ARIS – скриптов, например, импортирование организационной структуры из SAP HCM. В свою очередь, для поддержки актуальности данных в обоих системах реализован механизм экспорта данных из ARIS в Excel.

Таким образом, реализация справочных моделей позволяет:

  • владельцу модели централизовано управлять объектами предметной области в репозитории ARIS. Пример: изменение владельцем модели названия/типа объекта приведет к изменению данного названия/типа во всех процессах, использующих данный объект;
  • конечным пользователям выбирать информацию из имеющихся справочников для облегчения работы по описанию бизнес-процессов и экономии времени на поиск актуальной информации. Пример: использование объектов модели организационной структуры на моделях EPC;
  • вести структурированное описание деятельности организации посредством классификации моделей. Например, как правило, основу для классификации бизнес-процессов составляют четыре категории – основные, обеспечивающие, процессы управления и развития. Пример: основные процессы – маркетинг, обеспечивающие процессы – управление документами, процессы управления – бюджетирование, процессы развития – управление проектами;
  • разрабатывать референтные модели и тиражировать их на филиалы и дочерние общества организации, а также в иную проектную деятельность. Пример: использование ролевой модели, разработанной на этапе реинжиниринга, в проектной деятельности по внедрению SAP;
  • рационально работать в репозитории моделей, используя готовые объекты справочных моделей с описанием, и устранить избыточность при многократном использовании объектов в моделях. Пример: использование объектов с дерева документов исключает консолидацию таких объектов как «Договор на поставку материалов» и «Договор поставки материалов»;
  • выделять типовые и повторяющиеся объекты, анализировать их на предмет частоты использования и семантического корреляционного сходства. Пример: использование дерева функциональности способствует мониторингу дублирующейся функциональности, реализованной в разных ИТ-системах, в следствии чего, позволяет сократить количество ИТ-систем и, как итог, сократить затраты организации.
